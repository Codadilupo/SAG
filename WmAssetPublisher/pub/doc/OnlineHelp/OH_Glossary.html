A

Access Control Lists (ACLs)
A table that tells a computer operating system which access rights each user has to a particular system object, such as a directory or individual file. Each object has a security attribute that identifies its access control list. The list has an entry for each system user with access privileges. The most common privileges include the ability to read a file (or all the files in a directory), to write to the file or files, and to execute the file (if it is an executable file, or program).
Activation ID
A unique identifier that the Integration Server assigns to all documents published as a result of the one-time execution of the integration solution. The activation ID is stored in the activation field of a document envelope.
Adapter Connection
An object that contains parameters that adapter services and listeners use to connect to a resource. For information about creating a connection, refer to the documentation provided with the adapter. See also: Adapter notification and Listener.
Adapter notification
Determines whether an event has occurred on the adapter's resource and then sends the notification data to the Integration Server in the form of a published document. There are two types of adapter notifications: Polling notifications, which poll the resource for events that occur on the resource, and Listener notifications, which work with listeners to detect and process events that occur on the adapter resource. See also: Adapter Connection and Listener.
Adapter resource
The application or system with which an adapter provides connectivity and functionality; for example, SAP, PeopleSoft, Siebel, a database, etc.
Adapter service
A type of service that connects to an adapter’s resource and initiates an operation on the resource. Adapter services are created using service templates included with the adapter. See also: Adapter Connection and Adapter resource.
Alias
A shorthand method of identifying and describing a resource needed by the Integration Server. The Integration Server uses aliases to identify remote Integration Servers, databases, database drivers, database connection pools, and PKI profiles. The alias provides location, connection, and other information that the Integration Server needs to connect to the resource. See also: Driver Alias, Functional Alias, Pool Alias, PKI Profile Alias, Remote Server Alias.
Alert
The sending of information to designated users to indicate that some user action is required.
All (AND)
A join type that specifies that the Integration Server invokes the trigger service after it receives each document specified in the trigger condition. This is the default join type.
Any (OR)
A join type that specifies that the Integration Server invokes the trigger service when it receives any of the documents in the trigger condition.
API
Application program interface. An API is the specific method prescribed by a computer operating system or by another application program by which a programmer writing an application program can make requests of the operating system or another application.
An API can be contrasted with a graphical user interface or a command interface (both of which are direct user interfaces) as interfaces to an operating system or a program.
Archive
A set of data that has been saved in order to preserve a history of events or actions taken.
Asynchronous request/reply
A variation of request/reply in which the publishing flow service continues executing the steps in the service after publishing the document and executes another service to retrieve the reply.
Attribute declaration
Associates an attribute name with a simple type definition. An attribute declaration can specify a default value, a fixed value, and whether the appearance of the attribute in the instance document is required. Like element declarations, attribute declarations can be global or local.
Attribute reference
A reference from a complex type definition to a globally declared attribute. DTDs do not have attribute references. Consequently, attribute references do not appear in schemas generated from DTDs. For details, see the webMethods Integration Server Schema Reference.
At-least-once processing
A type of guaranteed document processing where the trigger processes the document once, but may also process duplicates of the document. The Integration Server provides at-least-once processing for guaranteed documents.
At-most-once processing
A type of document processing where document processing is attempted, but is not guaranteed. In at-most-once processing, the trigger processes a document once or not at all. The Integration Server provides at-most-once processing for volatile documents.
Audit database
The database the Integration Server uses to store logging, status, and other system information.
Audit
The ability to query, view, and report on logged information.

BRANCH
A flow step that executes a specified step based on the value of a variable in the pipeline.
Breakpoint
A feature used when testing and debugging to indicate a point in a flow service where you want processing to stop. Breakpoints can help you isolate a section of code or examine data values at a particular point in the execution path.
Broker
The globally scalable messaging backbone of a webMethods solution. It provides the infrastructure for implementing asynchronous, message-based solutions that are built on the publish-and-subscribe model or one of its variants, request/reply or publish-and-wait.
The role of the Broker is to route documents between information producers (publishers) and information consumers (subscribers). The Broker receives, queues, and delivers documents.

Cache Manager
The portion of the Integration Server that sends the results of a service to a local cache. Sending the results of a service to a cache can greatly improve throughput when many clients are requesting the same service with identical inputs.
Call stack
A feature used when testing and debugging that identifies which service generated the error and lists its antecedents. The call stack is LIFO based. That is, the top entry in the stack identifies the last (most recent) service invoked. The bottom entry identifies the first service that was invoked in a nested series.
Canonical document
A standardized representation that a document might assume while it is passing through webMethods components. A canonical document acts as the intermediary data format between resources.
Client classes
webMethods client classes are the thin client classes for Java, C/C++, and COM (Visual Basic, etc.) that enable secure, guaranteed messaging between partner applications across corporate firewalls. webMethods Client Classes communicate via XML-PRC over HTTP.
Clustering
A method of deploying multiple Integration Servers as if they were a single virtual server.
COM
Microsoft's framework for developing and supporting program component objects. It is aimed at providing similar capabilities to those defined in CORBA (Common Object Request Broker Architecture), a framework for the interoperation of distributed objects in a network that is supported by other major companies in the computer industry. Whereas Microsoft's OLE provides services for the compound document that users see on their display, COM provides the underlying services of interface negotiation, life cycle management (determining when an object can be removed from a system), licensing, and event services (putting one object into service as the result of an event that has happened to another object).
COM includes COM+, DCOM, and ActiveX interfaces and programming tools.
Commerce XML (cXML)
cXML is a set of lightweight XML DTDs based on the World Wide Web Consortium's XML standard, with the W3Cs associated request/response processes. cXML provides an infrastructure that streamlines the process of digitally exchanging catalog content and transactions in a secure manner.
Complex type definitions
A datatype definition in an XML Schema that describes the structure and content for an element. A complex type definition can contain element declarations, attribute declarations, and element references.
Concurrent document processing
A document processing mode in which the Integration Server processes the documents in the trigger queue in parallel.
Condition
An expression applied to a link between variables or used as the label for a BRANCH child. The Integration Server executes the link or the BRANCH child step only if the expression evaluates to true. See also Trigger condition.
Constraining facets
Constraints that limit the permitted values of a variable. The constraining facets you can use depend on the content constraints applied to the variable. Constraining facets and content constraints are applied to variables for validation purposes. See also: Content constraints.
Content constraints
Validation constraints that describe the types of data and possible values for a variables. For each content constraint, you can also specify constraining facets to limit the possible values for the variables. You can apply a content type that corresponds to one of the simple types built-in to XML or to a simple type defined in an XML Schema definition. See also Constraining facets and Simple type definition.
Content Manager
The portion of the Integration Server that is responsible for decoding client requests and encoding data that is returned to clients.
Context Manager
The portion of the Integration Server that allows the server to process multiple, concurrent requests using a single copy of the service in memory.
control step
The term used to collectively refer to the group of flow operations that provide conditional processing capabilities and allow you to build looping, branching, and retry structures in a flow service. Examples of flow-control steps include LOOP, BRANCH, REPEAT, and SEQUENCE.
CORBA
Common Object Request Broker Architecture.
 
D
          
Go to top
Data transformation
The changes made to variables and variable contents to resolve name, structure, and value discrepancies.
Data types
A classification of a type of information. Each data type supported by the Integration Server corresponds to a Java data type and has an associated icon.
Default document store
An in-memory document store that contains published documents delivered to the Integration Server directly. The dispatcher routes documents from the default document store to the trigger queues for subscribing triggers.
Dependent
An element that uses a selected element. For example, suppose that the flow ServiceA invokes the built-in service pub.xml:loadXMLNode. ServiceA uses the pub.xml:loadXMLNode service; therefore, flow ServiceA is a dependent of pub.xml:loadXMLNode.
Developer
Developer is a graphical development tool that you use to build, edit, and test integration logic. It provides an integrated development environment in which to develop the logic and supporting objects that carry out the work of an integration solution. It also provides tools for testing and debugging the solutions you create.
Dimensionality
The number of arrays to which a variable belongs. For example, the dimensionality of a single String is 0, that of a String list or document list is 1, and that of a String table is 2. A String that is a child of a document list has a dimensionality of 1. A String list that is a child of a document list has a dimensionality of 2.
Document
Objects that webMethods components use to encapsulate and exchange data. A document represents the body of data that a resource passes to the Integration Server. Often it represents a business event such as placing an order (purchase order document), shipping goods (shipping notice), or adding a new employee (new employee record).
Document Object Model (DOM)
An addressing system for accessing information within documents.
Document processing
The process of evaluating documents against specified trigger conditions and executing the appropriate trigger services to act on those documents.
Document stores
Saves published documents to disk or to memory while the documents wait for delivery or processing.
Document type
A named schema-like definition that describes the structure of a particular kind of document.
Document Type Definition (DTD)
Defines the SGML syntax dialect tags that are used to mark up a document. Tags within the HTML DTD are limited to the HTML standard; however, XML is a meta-language that can define new DTDs as needed. For example, XML can be used to describe a part in inventory with customized tags and/or attributes such as "Part Name", "SKU", and "Quantity."
Document validation
The process of verifying the structure and content of an individual document in the pipeline by validating it against a document.
Driver Alias
An alias for the database driver that the Integration Server uses for the webMethods audit database. The Integration Server uses the audit database to store logging, status, and other system information. See also: Functional Alias and Pool Alias.
Duplicate detection
A facility of the Integration Server that determines whether the current document is a copy of one previously processed by the trigger.

Electronic Data Interchange (EDI)
A system of syntax and semantics for the electronic exchange of business documents between trading partners. EDI has been in development and use for over 30 years by major corporations, first in North America and subsequently across the world. The syntax is not based on any existing Internet or Web syntax, because it predates both.
Element
An item that exists in the Navigation panel in webMethods Developer. Elements include folders, services, specifications, IS document types, triggers, and IS schemas. In the Navigation panel, servers and packages are not considered to be elements.
Element declarations
Associates an element name with a type definition. Like attribute declarations, element declarations can be global or local. The element declaration can contain attributes to specify a default value, a fixed value, and whether the element is abstract or nillable. If an element declaration is part of a content specification, the element declaration can contain attributes to specify occurrence constraints. For details, see the webMethods Integration Server Schema Reference.
Element references
A reference from an element declaration in a content specification to a globally declared element. For details, see the webMethods Integration Server Schema Reference.
E-mail Port
A port that accepts requests for services through email messages from a POP3 or IMAP mail server.
Envelope field
A field in a published document that contains information such as the sender’s address, the time the document was sent, the document UUID, and other useful information for routing and control. Each publishable document type uses a document reference to define the structure and content of the envelope.
Event
A specific action that the Event Manager recognizes and an event handler can react to. Event types include audit events, exception events, guaranteed delivery events, and replication events.
Event handler
A service that you write to perform some action when a particular event occurs. An event handler can be any type of service: a flow service or a Java service. Event handlers subscribe to the events that they want to be notified of. Use the Event Manager to subscribe an event handler to an event.
Exactly-once processing
A type of guaranteed document processing where the trigger process a document only once. The trigger does not process duplicates of the document. The Integration Server provides exactly-once processing for guaranteed documents received by triggers that have exactly-once properties enabled.
EXIT
A flow step that exits an entire flow service or a single flow step.
Explicitly linked variables
Pipeline variables and service variables that you manually link to each other. Explicit linking accomplishes the name and structure transformations necessary for a flow service. The Pipeline tab connects explicitly linked variables with a black line.
EXtensible Stylesheet Language (XSL)
A language for expressing stylesheets that defines how to present XML. Given a class of structured documents or data files in XML, designers use XSL to express their intentions about how that structured content should be presented; that is, how the source content should be styled, presented, and paginated onto some presentation medium such as a window in a Web browser or a set of physical pages in a book.

Filter
An expression that specifies content criteria for a published document. If a document does not meet the criteria specified in the filter, the document is not passed on to the trigger for processing. Filters are saved with the document subscription.
File Polling Port
A port that polls a monitoring directory for the arrival of files and then performs special processing, such as conversion or validation, on them. When the Integration Server detects a file in the directory, it runs the service that the port has been configured to use. This is the only service that can be invoked through this port.
Flat file dictionary
A collection of record definitions, field definitions, and composite definitions that can be used in multiple flat file schemas. For more information, see the Flat File Schema Developer’s Guide.
Flat file schema
A blueprint that contains the instructions for parsing or creating the records in a flat file, as well as the constraints to which a inbound flat file document should conform to be considered valid. Using flat file schemas, you can translate documents into and from flat file formats.
Flow step
The basic unit of work within a flow service. Flow steps are expressed in webMethods flow language and are interpreted and executed by the Integration Server at run time. Examples of flow steps include INVOKE, LOOP, and MAP.
Flow service
A service that is written in webMethods flow language. A flow service invokes other services and directs the flow of data among them. A flow service can include control steps that perform looping, branching, and retry processing.
FTP Port
A port that accepts requests for services through the File Transfer Protocol (FTP).
FTPS Port
A port that accepts requests for services through FTPS (FTP over SSL). The port uses Secure Sockets Layer (SSL) to authenticate and secure communications between the FTP client and the server.
Functional Alias
An alias that tells the Integration Server where in the webMethods audit database to write data for the webMethods functions Key Cross Referencing, Echo Suppression, Logging, and Document History. A functional alias also tells the Integration Server where to write Trading Networks information in the Trading Networks database. You cannot add or delete functional aliases, but you can change the JDBC connection pool to which a functional alias is assigned. See also: Driver Alias and Pool Alias.

Guaranteed delivery
A facility of the Integration Server that ensures guaranteed, one-time execution of services. It protects transactional requests from transient failures that might occur on the network, in the client, or on the server.
Guaranteed document delivery
A type of document delivery that ensures that a document, once published, is delivered at least once to the subscribing triggers. Guaranteed document delivery is provided for guaranteed documents.
Guaranteed processing
A type of document processing where the Integration Server ensures that a trigger makes at least one attempt to process a document. At-least-once processing and exactly-once processing are types of guaranteed processing. See also: At-least-once processing and Exactly-once processing.
Guaranteed storage
A type of document storage in which documents are saved to disk. During transport, guaranteed documents are acknowledged by a receiving resource before being removed by the sending resource. This allows a guaranteed document to be recovered if the resource on which the document resides, shuts down. The Integration Server provides guaranteed processing (at-least-once or exactly-once processing) for guaranteed documents.

Hypertext Markup Language (HTML)
The language that defines a simple, fixed type of document with markup designed for standard features such as headings, paragraphs, lists, and illustrations, with some limited provision for multimedia.
Hypertext Transfer Protocol (HTTP)
The protocol used to access hypertext and hypermedia information.
Hypertext Transfer Protocol/Security (HTTP/S)
The secure version of the Hypertext Transfer Protocol (HTTP).

IData object
The collection of name/value pairs on which a service operates. An IData object can contain any number of elements of any valid Java objects, including additional IData objects and IDataCodable objects.
IDE
Integrated Development Environment.
Implicitly linked variables
Pipeline variables and service variables the Pipeline tab automatically links (maps) to each other. The Pipeline tab automatically links variables whose names are the same and data types are compatible. The Pipeline tab connects implicitly linked variables with a gray line. The Pipeline tab does not display implicit maps if a MAP step is selected in the editor.
Input/Output validation
The process of verifying the inputs and/or outputs of a service by validating the inputs and/or outputs against the service signature.
Integration Server
The Integration Server is the webMethods Integration Platform’s central run-time component. It serves as the entry point for the systems and applications that you want to integrate, and is the platform’s primary engine for the execution of integration logic. It also provides the underlying handlers and facilities that manage the orderly processing of information from resources inside and outside the enterprise. The Integration Server publishes documents to and receives documents from the Broker.
INVOKE
A flow step that executes a specified service.

Java service
A service that is written in Java.
Java Database Connectivity (JDBC)
A Java API that enables Java programs, such as those used by the Integration Server, to execute SQL statements.
Java Naming and Directory Interface (JNDI)
An API that provides naming and directory functionality to applications written in the Java programming language. The JNDI is defined to be independent of any specific directory service implementation. Thus, a variety of directories (new, emerging, and already deployed ones) can be accessed in a common way.
JDBC
See Java Database Connectivity (JDBC).
JDBC connection pool
A predefined pool of connections that the Integration Server uses to connect to its audit database. See also: Audit database.
JNDI
See Java Naming and Directory Interface (JNDI).
Join condition
A trigger condition that associates two or more document types with a single service. Typically, join conditions are used to combine data published by different sources and process it with one service. See also: All (AND), Any (OR), and Only one (XOR).

Lightweight Directory Access Protocol (LDAP)
A standard for communicating with existing infrastructures that control and manage access to phone numbers, locations, passwords, access privileges, and anything else relevant to each directory entry.
Link
In the Pipeline tab, the line connecting variables linked to each other. If the variables are implicitly linked, the line is gray. If the variables are explicitly linked, the line is black. If properties have been set for the link (either an applied condition or an index value), the line is blue.
Listener
The component of an application that monitors a port for incoming requests for service. When a request arrives, the listener accepts the connection and sends the data to the appropriate component for processing. The Integration Server provides an HTTP Listener, an HTTPS Listener, and an FTP Listener; other listeners can be configured to listen on specified ports.
Listener notification
A type of adapter notification that works with listeners to detect and process events that occur on an adapter resource and then publishes the notification data in the form of a document.
Local name
The part of a universal name that uniquely identifies this service within the collection encompassed by the Namespace name. The name can be composed of any combination of letters, digits, or the period (.), dash (-), or underscore (_) characters. The name must begin with a letter or the underscore character. See also Universal name and Namespace name.
Local publish
A publishing scenario in which documents are published and processed within a single Integration Server. In a local publish, documents are not routed through a Broker.
Logging
The recording to persistent storage of all significant actions taken by the system to complete a transaction; also known as the audit trail.
LOOP
A flow step that executes a set of steps once for each element in a specified array.

MAP
A flow step that performs specified editing operations on the pipeline; for example adding variables to or dropping variables from the pipeline. A MAP step can also invoke transformers to perform value transformations in pipeline data.
Middleware
Software that connects two otherwise separate applications.

Name transformation
A data transformation in which the value and position of a variable in the document structure remains the same, but the name of the variable changes.
Namespace name
The part of a universal name that specifies the name used to qualify the local name of a service. The namespace name you specify must be a valid absolute URI (relative URIs are not supported). See also: Universal name and Local name.
Network Information Service (NIS)
A directory service available on the Solaris operating system that is used to store system-related information such as that relating to machines, networks, printers, and users.

Once-and-only-once processing
See Exactly-once processing.
Only one (XOR)
A join type that specifies that the Integration Server invokes the trigger service when it receives any of the document s specified in the trigger condition. For the duration of the join time-out period, the Integration Server discards documents if the documents were specified in the trigger condition and the documents have the same activation ID as the first document.
Outbound document store
A document store that contains published documents waiting to be sent to the Broker. The Integration Server places documents in the outbound document store when the configured Broker is not available. The Integration Server sends documents from the outbound document store to the Broker when the connection to the Broker is restored.

Go to top
Package
A container that is used to bundle services and related elements, such as specifications, IS document types, IS schemas, and output templates. When you create an element, you save it in a package.
Package replication
A process used to copy a package from one Integration Server and install it on another. See also: Replication service.
Partner profile
A set of information about trading partners that helps determine how and why the partners work together; it also helps companies with many partners to distinguish between partners.
Parsing
The operation that the Integration Server performs to convert an XML document (a String) into an XML node whose elements can be addressed and extracted by services. The Integration Server automatically parses XML documents that you fetch with the loadXMLNode service.
Partners
Companies that do business with one another.
Pipeline
The general term used to refer to the data structure in which input and output values are maintained for a flow service at run time. The pipeline starts with the input to the flow service and collects inputs and outputs from subsequent services in the flow. When a service in the flow executes, it has access to all data in the pipeline.
Pipeline tab
The facility you use to inspect the pipeline and route input and output between services in a flow.
Pipeline reference
A transformer or pipeline modifier (Link, Drop, or Set Value) involving a variable in a document reference. For example, if ServiceA declares a document reference to PODocument as a variable in its input signature, and ServiceA links a variable from PODocument to a new variable in the pipeline, then that link is a pipeline reference.
Pipeline validation
The process of verifying the structure and content of the pipeline by validating it against a document type.
PKI (Public Key Infrastructure) Profile
If you use PKI, this file contains your private key, user certificate, digital signature, CA certificate, certificate histories, and other information. You can store PKI profiles in your file system (as .epf files), or for even greater security, on an HSM device. See also: PKI Profile Alias.
PKI Profile Alias
If you use PKI, you assign this alias to a PKI Profile. For each profile, the alias specifies the location of the profile, defines ACL (Access Control List) associations, and provides a list of trusted certificates. See also: PKI (Public Key Infrastructure) Profile.
Polling notification
A type of adapter notification that polls the resource for events that occur on the resource and then sends the notification data to the Integration Server in the form of a published document.
Pool Alias
An alias for a JDBC connection pool. A JDBC connection pool is a predefined pool of connections that the Integration Server uses to connect to its audit database. Each Integration Server function (Key Cross Referencing, Echo Suppression, Logging, Document History, and Trading Networks) requires its own connection pool. Each pool alias specifies connection information for the database to which the pool is to connect, such as the database location, the number of connections to maintain, and idle timeout settings. See also: Driver Alias, Functional Alias.
Port
The endpoint to a logical connection on a server. When using TCP/IP, the port identifies a particular server program on a computer to client programs on other computers. The port is identified by a port number from 0 to 65535. Ports 0 to 1024 are reserved for use by certain privileged services (for example, port 80 is the default port for HTTP). The webMethods Integration Server is usually associated with port number 5555.
Prefetch
A property that specifies whether or not you want the Integration Server to automatically refresh the cache for a service when the cache expires. You can enable or disable the prefetch setting for a service using webMethods Developer.
Primary port
An HTTP or HTTPS port that you designate as the Integration Server's main listening port. The Integration Server does not use this port any differently than other HTTP or HTTPS ports, except that it doesn't allow a primary port to be deleted. This guarantees that at least one port is always available.
Primitive type
The data type of the variable when all dimensionality is removed. For example, the primitive type for a String list or a String table would be String. In XML Schema, a primitive data type is a fundamental data type from which other data types are derived.
Protocol Manager
The portion of the Integration Server that manages incoming requests for services. The webMethods Integration Server architecture is extensible and supports a number of simultaneous protocols including HTTP, SAP RFC, FTP, etc.
Proxy port
(Reverse Invoke configurations only) The port through which a Reverse Invoke Integration Server listens for requests from external clients. An Integration Server must have an enabled proxy port in order to function as a Reverse Invoke Integration Server. See also: Reverse Invoke, Reverse Invoke Integration Server.
Publishable document type
An IS document type with specific publishing properties. Instances of publishable document types can be published and subscribed to. Publishable document types can be used anywhere an IS document type is called for. Services can publish instances of publishable document types locally or to a Broker. Triggers subscribe to and process publishable document types.
Publishable document type for an adapter notification
An adapter notification can have an associated publishable document type that the adapter uses to send the notification data to an Integration Server or a Broker.
Publish-and-subscribe
A specific type of message-based solution in which messages are exchanged anonymously through a message broker. Applications that produce information that needs to be shared will make this information available in specific types of recognizable documents that they publish to the message broker. Applications that require information subscribe to the document types they need.
Purging
The removal of data stored at defined intervals via user action or scheduled job. Purging can be to another file or further storage, or it could complete a deletion.

Reference
A element that is used by a selected element. For example, suppose that the flow service ServiceA invokes the services pub.xml:loadXMLNode, ProcessPO, and SubmitPO, and declares a document reference to PODocument as part of its input signature. The services pub.xml:loadXMLNode, ProcessPO, and SubmitPO, and the document reference PODocument, are used by ServiceA; therefore, they are references of ServiceA.
Registration Port
(Reverse Invoke configurations only) The port through which a Reverse Invoke Integration Server maintains its connection to an associated internal server. The internal Integration Server and the Reverse Invoke Integration Server communicate with each other through this port using one of two proprietary, asynchronous communication protocols: SOCK, which is unencrypted and uses plain TCP sockets, and SSLSOCK, which is encrypted and uses SSL sockets. These connections are persistent. See also: Reverse Invoke, Reverse Invoke Integration Server, SOCK, SSLSOCK.
Remote Server Alias
An alias that describes how your Integration Server connects to another webMethods Integration Server.
REPEAT
A flow step that re-executes a set of steps up to a specified number of times based on the successful or unsuccessful completion of the set.
Replication service
A service that the Integration Server automatically executes when it prepares to replicate a package. Replication services provide a way for a package to persist state or configuration information so that these are available when the published package is activated on the remote server.
Repository Server
The mechanism the Integration Server uses to store data in and retrieve data from the repository. The repository manages configuration and transaction information for all clustering functions and some Broker functions.
Request/reply
A variation of the publish-and-subscribe model in which an application requests information and waits for a response. The publisher can broadcast the request to all subscribers or deliver the request to a specific subscriber. A request/reply can be asynchronous or synchronous. See also Asynchronous request/reply and Synchronous request/reply.
Reverse Invoke
A configuration that allows your internal Integration Server to operate in environments where connections from a DMZ to the internal network are prohibited. In this configuration, your internal server remains behind your inner firewall, where external clients cannot access it. Another Integration Server, called a Reverse Invoke Integration Server, resides in your DMZ and acts as an intermediary between the Internet and your internal server. See also Reverse Invoke Integration Server.
Reverse Invoke Integration Server
A specially configured Integration Server that resides in your DMZ and protects your internal Integration Server by intercepting requests from external clients. For an Integration Server to function as a Reverse Invoke Integration Server, it must have a proxy port to listen for requests from external clients (partners) and a registration port through which it maintains its connection to the internal server. See also Reverse Invoke, Proxy port, Registration Port.
Reverse Proxy Alias
An alias assigned to a Reverse Invoke Integration Server. The alias contains connection information such as host name or IP address and communications protocol. See also Reverse Invoke, Reverse Invoke Integration Server.
Routing signature
The routing-related fields in a document header (sender, receiver, document type, etc.) that a specific partner management implementation cares about.

schema
The blueprint or model document that you validate an XML document against. The schema defines what can and cannot be contained in the XML documents being validated.
Secure Sockets Layer (SSL)
A program layer created by Netscape for managing the security of message transmissions in a network. This program layer separates the code that ensures message confidentiality for an application (such as your Web browser or HTTP) from the Internet's TCP/IP layers. The "sockets" part of the term refers to the sockets method of passing data back and forth between a client and a server program in a network or between program layers in the same computer. The Netscape SSL uses the public-and-private key encryption system from RSA, which also includes the use of a digital certificate.
Security Manager
The portion of the Integration Server that provides security measures for:
Authentication. The Integration Server authenticates all client connections by validating a user ID and password or by using a digital certificate. The Integration Server authenticates itself to other clients or remote webMethods Integration Servers by using its own digital certificate.

Authorization. The Integration Server uses Access Control Lists (ACLs) to define what users (or groups of users) are authorized or denied access to a service or a folder that includes a collection of services. Data Privacy uses the Secure Socket Layer protocol (SSL) to secure both inbound and outbound requests. SSL libraries are embedded in the webMethods Integration Server and in the webMethods Client libraries.
SEQUENCE
A flow step that groups a set of steps into a series. The SEQUENCE step is implicit in most cases where a group of flow steps are listed one after another; however, it may be used explicitly to override default operating behavior and  specify the condition on which the sequence exits.
Serial document processing
A document processing mode in which Integration Server processes the documents in the trigger queue one after the other. Serial document processing maintains publication order because it processes documents in the same order in which it retrieves the documents from the Broker.
Service
Services are method-like units of work that operate on documents. They contain logic that the Integration Server executes. You build services to carry out work such as extracting data from documents, interacting with back-end resources, and publishing documents to the Broker. When you build a trigger, you specify the service that you want to use to process the documents that you subscribe to.
Shutdown service
A service that the Integration Server automatically executes when it unloads a package from memory. Shutdown services are useful for executing clean-up tasks such as closing files and purging temporary data. You can also use them to capture work-in-progress or state information before a package unloads.
Signature
A term that is often used to refer to the set of variables (names and data types) that a service takes as input and produces as output. For example, the signature of a service that generates an authorization code for a credit card might consist of two string inputs (AcctNumber and ExpirationDate) and one string output (AuthCode).
Simple condition
A trigger condition that associates one publishable document type with a service. When the Integration Server receives instances of that publishable document type, it will execute the specified service.
Simple type definition
A datatype definition in an XML Schema that describes or limits the values for an element. Simple type definitions cannot have element content and cannot carry attributes.
Specification
A formal description of a service’s inputs and outputs that you create when you build a service. By including a specification, you make the input and output requirements of your service known to other developers who may want to use that service in a flow. It also enables a service's inputs and outputs to be recognized by webMethods Developer so they can be mapped with values in the pipeline.
SOCK
A proprietary, asynchronous communication protocol that an internal server uses to connect to a Reverse Invoke Integration Server in the DMZ. SOCK is unencrypted and uses plain TCP sockets. The connection is persistent. See also: Reverse Invoke Integration Server, Secure Sockets Layer (SSL),and SSLSOCK.
SSLSOCK
A proprietary, asynchronous communication protocol that an internal server uses to connect to the Reverse Invoke Integration Server in the DMZ. SSLSOCK is encrypted and uses SSL sockets. The connection is persistent. See also: Reverse Invoke Integration Server, Secure Sockets Layer (SSL), and SOCK.
Standard Generalized Markup Language (SGML)
A standard formatting language that enables a publisher to create a single document source that can be displayed or printed in a variety of ways. SGML is a large, complex formatting language that is not especially suited to the Web. HTML and XML are simplified versions of SGML that are specifically designed to support Web applications.
Startup service
A service that the Integration Server automatically executes when it loads a package into memory. Startup services are useful for generating initialization files or assessing and preparing the environment before the server loads a package.
Structural constraints
In pipeline validation, structural constraints specify the existence and structure of variables at run time. In XML validation, structural constraints describe the structure and sequence of elements in the XML document.
Structural transformation
A data transformation in which the value of the variable remains the same, but the data type or position of the variable in the document structure changes.
Synchronous request/reply
A variation of request/reply in which the publishing service stops executing while it waits for a response. When the service receives a reply document, the service resumes execution.

Threads
The basic unit of program execution. A process can have several threads running concurrently, each performing a different job. When a thread has finished its job, the thread is suspended or destroyed. When a webMethods client invokes a service, that service runs as a thread within the Integration Server program.
Thread pool
A group of threads available for executing services. The Integration Server has two thread pools: the server thread pool and the scheduler thread pool. The server thread pool controls how many threads the server uses to execute services, retrieve documents from the Broker, and execute triggers. The scheduler thread pool controls how many threads the server uses to execute scheduled services.
Transformer
A service you insert into a MAP step to accomplish a value transformation. Any service can be inserted as a transformer. By inserting transformers into a MAP step, you can accomplish your end-to-end mapping in a single view.
Trigger
An element that establishes subscriptions to publishable document types and specifies how to process instances of those publishable document types.
Trigger condition
The fundamental component of a trigger that associates one or more publishable document types with a single service. The trigger subscribes to all of the publishable document types listed in a trigger condition. When the trigger receives documents to which it subscribes, the Integration Server processes the document by invoking the specified service. Triggers can contain simple conditions or join conditions. See also: Simple condition and Join condition.
Trigger document store
A document store that contains documents waiting to be processed by triggers. The Integration Server assigns each trigger a queue in the trigger document store.
Trigger queue
The portion of the trigger document store allotted to a trigger.
Trigger service
The service that processes a document received by a trigger. Each trigger condition specifies a single trigger service.

Universal name
A unique public identifier that external protocols (such as SOAP) use to reference a service on a webMethods Integration Server. Every service on a webMethods Integration Server has a universal name in addition to its regular webMethods name. A universal name has two parts: a namespace name and a local name. See also Namespace name and Local name.
Universally unique identifier (UUID)
A unique identification number assigned to a document during publication. The UUID is used by subscribers of the document to perform duplicate detection.
User notification
The sending of information to designated users to indicate that some user action is required; also called an alert.

Validation
The process of verifying the structure and/or content of an object by matching it against a model or blueprint. If the object does not conform to the model or blueprint, it is invalid. See alsoInput/Output validation, Pipeline validation, Document validation, and XML validation.
Value transformation
A data transformation in which the name and position of the variable remain the same, but the data contained in the variable changes. (For example, you can change the format of a date, concatenate two strings, or add the values of two variables together.)
Values object
In earlier versions of the webMethods Integration Server (prior to 3.5), the universal container that services use to receive input from and deliver output to programs that invoke it. The Values object is an ordered hash table that contains the individual data elements upon which a service operates. It can contain any number of elements of any valid Java data type, including additional Values objects. For current versions of the webMethods Integration Server, see IData object.
Volatile storage
A type of document storage in which documents are saved in memory. Volatile documents are never acknowledged by the receiving resources. Volatile documents will be lost if the resource on which the volatile document resides fails or shuts down. Delivery and processing are attempted but not guaranteed for volatile documents.

A development environment for creating, maintaining, publishing and testing services. See also: Developer.
webMethods Integration Server
A secure, high-performance deployment environment that allows the automated exchange of data between applications, Web sites, and other data sources. See also: Integration Server.
WQL
webMethods Query Language (formerly, WOM). A language used to retrieve information from an HTML or XML document. See also XQL.

XML
EXtensible Markup Language, a uniform method for describing and exchanging data that is flexible, extensible, and easy to implement. It is a simplified dialect of the Standard Generalized Markup Language (SGML).
XML/EDI
EDI semantics in XML syntax. See also: Electronic Data Interchange (EDI)
XML-Encoded Remote Procedure Call (XML-RPC)
Allows XML DTDs to describe both the inputs and outputs to a service, and also the message format for the data that is sent to and returned from the Integration Server. XML-RPC is implemented on top of standard Web protocols, specifically HTTP, SSL, and X.509, and is used for thin clients to access the Integration Server.
XML node
An element-based representation of an XML document, which is the result of a parsing operation. The node expresses a document in a tree-like structure that allows the data within it to be efficiently addressed and linked into services. See also: Parsing.
XML Schema
A model or description of the structure of an XML document. In the Integration Server, you can create schemas from XML Schema definitions. See also: schema.
XML validation
The process of verifying the structure and content of an XML document by making sure the document conforms to a schema. See also: schema.
XQL
XML Query Language. A language used to retrieve information from an XML document. See also: WQL.
